# skill-proposal

A proposed change to an agent skill file, generated by the reflect loop
or by a human reviewer. Follows the "agent proposes, human accepts"
rule — the agent never edits skills directly. Each proposal cites the
learnings and artifacts that motivated it.

Accepted proposals apply a diff to the target skill file (committed via
git). Rejected proposals are archived with their reason, so future runs
of the `improve` skill don't re-suggest the same rule. `check_at` on an
accepted proposal schedules a future review: did this change actually
help? The `## Outcome` appended then closes the loop on the proposal
itself.

## Frontmatter

```yaml
title:        { type: string, required: true }
target_skill: { type: string, required: true }
status:       { type: enum, values: [pending, accepted, rejected], default: pending }
domain:       { type: string, required: false }
change_type:  { type: enum, values: [add, modify, remove], default: modify }
evidence:     { type: 'string[]', required: false }
check_at:     { type: date, required: false }
mentions:     { type: 'string[]', required: false }
```

## Status machine

```yaml
initial: pending
transitions:
  pending: [accepted, rejected]
```

## Required sections

On `status: accepted` → body must contain `## Outcome`
On `status: rejected` → body must contain `## Outcome`
